## Design System

| Property | Value |
|----------|-------|
| Tool | Manual Tailwind 4 token system |
| Preset | not applicable |
| Component library | Custom React primitives: `Section`, `Button`, `Card`, `RichText`, `StoryDisclosure`, `ProductForm`, `ProductGallery`, route-local App Router components |
| Icon library | `lucide-react` |
| Font | `font-display` for headings, `font-sans` for body and controls, `font-mono` for metadata |

Design-system source of truth:

- Use `src/app/globals.css` `@theme` tokens and existing utilities.
- Use `Section.Root` and `Section.Container` for storefront bands and page containment.
- Use `cn()` from `@/lib/utils` for all conditional class composition.
- Do not introduce shadcn, CSS modules, inline styles, raw color values, cool gray palettes, or new page-level visual systems for this phase.
- New shared UI components need co-located Storybook coverage; route-only SEO remediation should stay inline or in route-local `_components` files when reuse is not real.

shadcn gate result: `components.json` is absent, but the project already has an approved manual design system through Tailwind 4 tokens, shared primitives, and AGENTS.md rules. Phase 18 must use that system rather than initialize a new registry-backed one.

## Spacing Scale

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, breadcrumb separators, tight metadata spacing |
| sm | 8px | Compact offsets, link focus breathing room, small card gaps |
| md | 16px | Default element spacing, rich-text paragraph spacing, compact padding |
| lg | 24px | Hero intro separation, toolbar-to-grid rhythm, content group spacing |

Exceptions: none for new phase-owned spacing. Preserve existing responsive layout tokens such as `px-gutter`, `py-section`, `max-w-base`, `max-w-wide`, and existing primitive control heights instead of creating ad-hoc values.

## Typography

Declared roles for Phase 18 edits:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px via `type-body` | 400 | 1.6 |
| Label | 14px via `type-label` or `type-mono-meta` | 600 | 1.25 |
| Heading | 24px via existing heading utility | 600 | 1.2 |
| Display | 44px via existing route H1 utility | 600 | 1.04 |

Typography contract:

- Collection and product pages must render exactly one visible `h1` in crawlable HTML.
- Banner collection pages must show the collection title as a visible display heading below or adjacent to the banner artwork; `sr-only` cannot be the only H1 presentation.
- Imported Shopify/Sanity rich content must not emit `h1` or `h2` inside read-more, product description, review, related-product, or recommendation sections. Demote imported headings to lower-level content headings before rendering.
- Do not add new type sizes or weights for this phase. Use the four roles above and existing utilities only.
- Preserve text wrapping protections from globals: long SEO titles, product names, and imported rich text must wrap inside their parent containers on mobile and desktop.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-paper`, `text-ink` | Page backgrounds, collection/PDP base surfaces, long-form rich content |
| Secondary (30%) | `bg-card`, `bg-paper-2`, `border-hairline` | Product cards, disclosures, sidebars, structured evidence panels, subtle section separation |
| Accent (10%) | `bg-brand`, `text-brand`, `text-gold-deep`, `ring-ring` | Primary CTA, active breadcrumb/current-page marker, active filters, focus ring, key SEO hierarchy emphasis |
| Destructive | `text-danger`, `bg-danger-tint` | Not used in Phase 18 storefront UI unless an existing destructive account/cart pattern is touched |

Accent reserved for: primary CTA, active breadcrumb/current-page marker, active filters, visible focus states, and one hierarchy emphasis per SEO-critical surface. Do not use accent on every link, every card border, decorative SEO evidence notes, or broad rich-text content.

Color contract:

- Keep the warm/botanical palette. Do not introduce cool gray, blue, purple, or raw color utility drift.
- Collection remediation should make the H1 visible through hierarchy and contrast, not through a new accent palette.
- Metadata, robots, sitemap, and structured-data implementation work should not add visible storefront badges or debug labels.

## Visual Hierarchy and Interaction Contract

| Surface | Primary focal point | Contract |
|---------|---------------------|----------|
| Banner collection page | Visible collection H1, then brief intro, then product grid | Banner artwork may appear first, but title and intro must be readable, visible, and before product cards. Long story/read-more content moves below the product grid. |
| Default collection page | Collection H1 and intro inside existing brand/surface hero | Preserve breadcrumb, title, intro, toolbar, sidebar, and product-grid scan order. |
| Product page | Product title H1 and buy section | Product title is the only visible H1. Buy controls remain prominent before description; reviews and recommendations cannot outrank the product title. |
| Blog/tag indexation surfaces | Existing blog/listing hierarchy | No visual redesign. Tagged pages can change metadata/noindex behavior without adding visible warnings. |
| Service/contact/schema surfaces | Existing page hero and visible content | Only add schema where visible content supports it; do not create hidden schema-only UI claims. |

Interaction rules:

- Keep route components server-first. Push any browser behavior to existing interactive leaves.
- Preserve native `details` disclosure behavior for collection story content unless execution proves a route-local replacement is required.
- Icon-only actions must use existing accessible primitives and labels. New SEO remediation should prefer text or icon+text actions where the action is not universally recognizable.
- Loading states must not mask crawl-critical collection/PDP title, intro, product, metadata, canonical, or JSON-LD content in first-response HTML.

- Redirect, sitemap, Search Console, and production-host handoff notes belong in docs/evidence files, not visible storefront UI.

## Phase 18 Surface Contracts

### Collection Heading and Content Placement

- Banner collection pages must render breadcrumb, optional banner image, visible collection H1, short intro, toolbar/filter controls, product grid, then long story/read-more content.
- `StoryDisclosure` content that represents long SEO/read-more copy must render below the product grid in DOM order.
- Collection rich descriptions must be sanitized before rendering and must not introduce extra H1/H2 headings.
- Product-grid anchors, pagination, filters, and sidebar scan order must remain stable after moving long content.

### Product Page Heading and Crawlability

- PDP title is the only visible H1.
- Product description, specs, review widgets, related products, and recommendation rails must not emit H1.
- The buy section stays visually primary after the product title and rating summary.
- Raw HTML evidence must show meaningful product title, buy-section context, metadata, canonical, and JSON-LD before hydration.

### Metadata, Indexation, and Schema Work

- Title/canonical/robots/sitemap/blog-tag changes should be invisible to normal storefront users unless an existing visible route label already exists.
- Service, LocalBusiness, FAQ, Review, and Product schema must map to visible page content and reliable data. Do not invent visible or hidden claims to satisfy audit checklists.
- `/blog/` simplification, redirect handoffs, Search Console tasks, and final-host redirects belong in code/docs/evidence according to Phase 18 decisions, not in new storefront notices.

### Performance and Loading

- Avoid broad skeleton-only static shells for collection/PDP routes when the audit requires crawlable HTML proof.
- Preserve existing warm, restrained skeleton styling only for truly dynamic content below crawl-critical title/intro/product context.
- Any performance remediation that touches layout must keep text, buttons, cards, and images from overlapping across mobile and desktop viewports.
