## Route Structure

- Blog index route: `src/app/(storefront)/blogs/[blog]/page.tsx`.
- Tagged listing route: `src/app/(storefront)/blogs/[blog]/tagged/[tag]/page.tsx`.
- Article detail route: `src/app/(storefront)/blogs/[blog]/[article]/page.tsx`.
- Atom feed route: `src/app/(storefront)/blogs/[blog]/atom/route.ts`.
- Route props already use the Next.js 16 `params: Promise<...>` and `searchParams: Promise<...>` pattern.
- Listing/detail pages normalize legacy blog handle `journal` to canonical `teavision-blogs` through `normalizeBlogHandle()`, but there is no explicit redirect today.

## Current Data Source

- Blog data comes from Shopify Storefront GraphQL via `src/lib/blog/operations.ts`.
- GraphQL documents live in `src/lib/shopify/queries/blog.graphql`.
- `getBlog()` paginates all Shopify articles in batches of 250, reshapes each summary, sorts through Shopify `PUBLISHED_AT reverse: true`, and filters/paginates in memory.
- `getArticle()` fetches one Shopify article by blog handle and article handle.
- Both operations use Next Cache Components with `use cache`, `cacheTag('blog', ...)`, and `cacheLife('hours')`.

## Components and Layout

- Listing shell lives in `src/app/(storefront)/blogs/[blog]/_components/listing-page.tsx`.
- Listing content composes `FeaturedArticles`, `ArticleResults`, `TagFilterNav`, `ArticleList`, `Pagination`, `EmptyState`, and `NewsletterSignup`.
- Article detail renders its own route-level layout with breadcrumb, tag pills, title, metadata, featured image, excerpt card, rich text, previous/next articles, optional comments, and JSON-LD.
- Shared cards use `src/components/ui/article-card/article-card.tsx`.
- All styling uses Tailwind token classes and existing UI primitives. Continue this pattern.

## SEO, Metadata, Feeds, and Sitemap

- Listing metadata is generated in `src/app/(storefront)/blogs/[blog]/_lib/metadata.ts`.
- Article metadata is generated directly in `[article]/page.tsx`, including canonical URL, Open Graph article type, published time, and featured image.
- Article detail emits `BlogPosting` JSON-LD.
- `src/app/sitemap.ts` currently calls `getBlog(DEFAULT_BLOG_HANDLE)` to add blog index, article URLs, and tag URLs.
- Atom feed currently serializes all Shopify article summaries from `getBlog()`.

## Images and Rich Text

- Current article and listing images come from Shopify image objects and are allowed by `next.config.ts` remote pattern for `cdn.shopify.com`.
- Article rich text is Shopify HTML from `contentHtml`, sanitized through `sanitizeShopifyArticleHtml()`, then rendered by `RichText`.
- Shopify comments are also rendered as sanitized HTML if any exist.
- Sanity migration should replace article body rendering with Portable Text rendering, while leaving Shopify HTML sanitization untouched for product/page/collection content.

## Hardcoded Content

- `DEFAULT_BLOG_HANDLE = 'teavision-blogs'`, `LEGACY_BLOG_HANDLE = 'journal'`, `ARTICLES_PER_PAGE = 6`, and two `FEATURED_ARTICLE_HANDLES` are hardcoded in `src/lib/blog/operations.ts`.
- Blog hero title, hero image, and listing description are hardcoded in `src/components/blog/hero/hero.tsx` and `src/lib/blog/listing.ts`.
- Homepage Tea Journal teasers are hardcoded in `src/components/homepage/content.ts` and rendered by `src/components/homepage/tea-journal/tea-journal.tsx`.
- Header and footer links hardcode `/blogs/teavision-blogs`.

## Sibling CMS State

- `../teavision-cms` exists locally as a directory, but it is empty and not a Git repository in this workspace.
- Plan assumes the implementation will either initialize Sanity Studio there or connect it to an existing remote Sanity project.

## Dependencies and Assumptions

- Current storefront dependencies do not include `next-sanity`, `@portabletext/react`, or `@sanity/image-url`.
- Existing Shopify credentials are still required for catalog/cart/page data.
- New Sanity env vars should fail fast, following `shopifyFetch()` behavior, when required published-content config is missing.
- Use scripted migration as the default because image assets, slugs, SEO fields, dates, and body conversion need repeatability.
- Manual editorial QA remains required for body conversion fidelity, especially tables, embedded images, unusual HTML, and comments.
- Existing blog URLs must not break; redirects are only needed if old aliases or malformed paths are discovered.

## Document Types

### `blog`

Represents the blog container for `teavision-blogs` and keeps route-level editorial settings out of code.

Fields:

- `title` string, required.
- `slug` slug, required, source `title`, default current value `teavision-blogs`.
- `description` text, required, max 240.
- `heroImage` `imageWithAlt`, optional but recommended.
- `featuredPosts` array of references to `blogPost`, max 2 initially to preserve current layout.
- `seo` object, optional route-level fallback.

### `blogPost`

Represents a public article.

Fields:

- `blog` reference to `blog`, required.
- `title` string, required, max 120.
- `slug` slug, required, source `title`, unique within the selected blog.
- `excerpt` text, required, max 240.
- `featuredImage` `imageWithAlt`, required.
- `author` reference to `author`, optional with frontend fallback to `Teavision Team`.
- `categories` array of references to `category`, optional.
- `tags` array of strings, optional, normalized and deduplicated by migration.
- `publishedAt` datetime, required.
- `body` `blockContent`, required.
- `seo` object, optional per-post metadata.
- `legacy` object with `shopifyArticleId`, `blogHandle`, `articleHandle`, `sourceUrl`, `importedAt`, and optional `contentHtml` for audit-only fallback.
- `legacyComments` optional read-only array only if stakeholders confirm comments matter.

### `author`

Reusable editorial author.

Fields:

- `name` string, required.
- `slug` slug, source `name`.
- `image` `imageWithAlt`, optional.
- `bio` text, optional.

### `category`

Reusable category/taxonomy item.

Fields:

- `title` string, required.
- `slug` slug, required, source `title`, unique.
- `description` text, optional.

## Object Types

### `seo`

Reusable SEO object:

- `metaTitle` string, max 70, warning after 60.
- `metaDescription` text, max 170, warning after 155.
- `noIndex` boolean, default false.
- `canonicalPath` string, optional and restricted to site-relative `/blogs/...` paths.
- `ogImage` `imageWithAlt`, optional.

### `imageWithAlt`

Reusable image object:

- Sanity `image` with `options.hotspot = true`.
- `alt` string, required unless the field is explicitly marked decorative.
- `caption` string, optional.
- `attribution` string, optional.

### `blockContent`

Portable Text array:

- Standard `block` with styles `normal`, `h2`, `h3`, `h4`, `blockquote`.
- Marks `strong`, `em`, `underline`, and link annotations with URL validation.
- List support for bullet and numbered lists.
- Custom object-like blocks for `imageWithAlt` and optionally a simple `callout`.
- Avoid anonymous object types if future GraphQL exposure is likely.

## Schema Best-Practice Notes

- Use `defineType`, `defineField`, and `defineArrayMember` from `sanity`.
- Keep reusable objects in `schemaTypes/objects/` and documents in `schemaTypes/documents/`.
- Enable image hotspot/crop and query image asset metadata for dimensions/LQIP.
- Query slugs by `slug.current`.
- Keep migrated legacy identifiers in a nested `legacy` object so future migrations are idempotent.

Minimal implementation sketch:

```ts
import { defineArrayMember, defineField, defineType } from 'sanity'

export const blogPost = defineType({
  name: 'blogPost',
  title: 'Blog post',
  type: 'document',
  fields: [
    defineField({
      name: 'blog',
      type: 'reference',
      to: [{ type: 'blog' }],
      validation: (Rule) => Rule.required(),
    }),
    defineField({
      name: 'title',
      type: 'string',
      validation: (Rule) => Rule.required().max(120),
    }),
    defineField({
      name: 'slug',
      type: 'slug',
      options: { source: 'title', maxLength: 200 },
      validation: (Rule) => Rule.required(),
    }),
    defineField({
      name: 'excerpt',
      type: 'text',
      rows: 3,
      validation: (Rule) => Rule.required().max(240),
    }),
    defineField({
      name: 'featuredImage',
      type: 'imageWithAlt',
      validation: (Rule) => Rule.required(),
    }),
    defineField({
      name: 'author',
      type: 'reference',
      to: [{ type: 'author' }],
    }),
    defineField({
      name: 'categories',
      type: 'array',
      of: [
        defineArrayMember({ type: 'reference', to: [{ type: 'category' }] }),
      ],
    }),
    defineField({
      name: 'tags',
      type: 'array',
      of: [defineArrayMember({ type: 'string' })],
    }),
    defineField({
      name: 'publishedAt',
      type: 'datetime',
      validation: (Rule) => Rule.required(),
    }),
    defineField({
      name: 'body',
      type: 'blockContent',
      validation: (Rule) => Rule.required().min(1),
    }),
    defineField({ name: 'seo', type: 'seo' }),
    defineField({
      name: 'legacy',
      type: 'object',
      readOnly: true,
      fields: [
        defineField({ name: 'shopifyArticleId', type: 'string' }),
        defineField({ name: 'blogHandle', type: 'string' }),
        defineField({ name: 'articleHandle', type: 'string' }),
        defineField({ name: 'sourceUrl', type: 'url' }),
        defineField({ name: 'importedAt', type: 'datetime' }),
      ],
    }),
  ],
  preview: {
    select: { title: 'title', subtitle: 'publishedAt', media: 'featuredImage' },
  },
})
```

```ts
export const blogPostSummaryFields = `
  _id,
  title,
  "handle": slug.current,
  excerpt,
  publishedAt,
  tags,
  "categories": categories[]->{title, "slug": slug.current},
  "authorName": author->name,
  featuredImage{
    alt,
    caption,
    asset->{
      _id,
      url,
      metadata{dimensions, lqip}
    },
    crop,
    hotspot
  },
  seo
`

export const blogBySlugQuery = `
  *[_type == "blog" && slug.current == $blogHandle][0]{
    _id,
    title,
    "handle": slug.current,
    description,
    heroImage,
    seo,
    "featuredPostIds": featuredPosts[]._ref
  }
`

export const blogListingQuery = `
  {
    "blog": *[_type == "blog" && slug.current == $blogHandle][0]{
      _id, title, "handle": slug.current, description, heroImage, seo,
      "featuredPosts": featuredPosts[]->{
        ${blogPostSummaryFields}
      }
    },
    "articles": *[
      _type == "blogPost" &&
      blog->slug.current == $blogHandle &&
      publishedAt <= now()
    ] | order(publishedAt desc) {
      ${blogPostSummaryFields}
    },
    "categories": *[_type == "category"] | order(title asc) {
      title, "slug": slug.current
    }
  }
`

export const articleBySlugQuery = `
  *[
    _type == "blogPost" &&
    blog->slug.current == $blogHandle &&
    slug.current == $articleHandle &&
    publishedAt <= now()
  ][0]{
    ${blogPostSummaryFields},
    body,
    "blog": blog->{title, "handle": slug.current},
    "previous": *[
      _type == "blogPost" &&
      blog._ref == ^.blog._ref &&
      publishedAt < ^.publishedAt &&
      publishedAt <= now()
    ] | order(publishedAt desc)[0]{${blogPostSummaryFields}},
    "next": *[
      _type == "blogPost" &&
      blog._ref == ^.blog._ref &&
      publishedAt > ^.publishedAt &&
      publishedAt <= now()
    ] | order(publishedAt asc)[0]{${blogPostSummaryFields}}
  }
`
```

<migration_strategy>

Default strategy: scripted migration with editorial QA.

1. Initialize or connect `../teavision-cms` as a Sanity Studio project.
2. Add schemas and deploy them to a staging dataset first.
3. Export Shopify blog source through the existing Storefront API query shape, extending the export query only if missing fields are needed.
4. Write a migration script that:
   - Fetches every Shopify article for `teavision-blogs`.
   - Creates or upserts one `blog` document for `teavision-blogs`.
   - Creates or upserts authors from `authorV2.name`.
   - Creates or upserts categories/tags from Shopify tags after normalization.
   - Downloads Shopify CDN featured images and embedded article images, uploads them into Sanity assets, and preserves alt/caption data when available.
   - Converts `contentHtml` to Portable Text through an allowlisted transform.
   - Stores original `contentHtml` in a read-only legacy/audit field until QA passes.
   - Uses deterministic `_id` values based on Shopify IDs or blog/article handles so reruns are idempotent.
5. Run migration in dry-run mode first and write a JSON report with counts for posts, images, authors, tags/categories, conversion warnings, and skipped nodes.
6. Run migration against staging, QA the Studio content and frontend preview, then run against production.
7. Freeze Shopify blog publishing during final cutover, rerun the migration, compare article counts/slugs/date ranges, then deploy the frontend cutover.

Manual entry is not recommended except for fixing conversion warnings because it is harder to preserve dates, slugs, images, SEO fields, and idempotency.

</migration_strategy>

<risks_and_edge_cases>

- HTML-to-Portable-Text conversion can lose complex tables, inline styles, embeds, captions, or unusual Shopify markup. Mitigation: keep `legacy.contentHtml` temporarily, report unsupported nodes, and require editorial QA before deleting the Shopify source path.
- Shopify comments exist in the current route. Mitigation: confirm whether to migrate them as read-only `legacyComments`; otherwise document intentional removal.
- Existing tag URLs are generated by a custom `slugifyTag()` helper. Mitigation: preserve the helper and migrate category/tag slugs to match it.
- Sanity image migration can create duplicate assets on rerun. Mitigation: use deterministic metadata/source URL checks before upload or keep an asset lookup cache.
- Sanity CDN image host is not currently allowed in `next.config.ts`. Mitigation: add specific `cdn.sanity.io/images/<projectId>/<dataset>/**` remote pattern.
- Draft/preview support adds token and cache complexity. Mitigation: ship published-content migration first unless editorial preview is a launch requirement.
- The sibling CMS folder is empty locally. Mitigation: initialize Studio in that folder or clone/populate the real Studio project before implementing schemas.
- If Shopify blog publishing continues during migration, final article count can drift. Mitigation: freeze publishing during final cutover and rerun the migration immediately before deploy.
- `generateMetadata` with Cache Components needs cached data or intentional dynamic behavior. Mitigation: use the same cached Sanity operations as pages and avoid runtime-only APIs inside cached scopes.

</risks_and_edge_cases>

<rollback_strategy>

- Keep Shopify-backed blog operations available on a short-lived rollback branch or behind an internal `BLOG_SOURCE=shopify|sanity` env switch until Sanity production content is accepted.
- Do not delete Shopify blog content during the cutover.
- Keep migration reports and Sanity document IDs so bad imports can be deleted by `_id` prefix or restored from a dataset export.
- Before production migration, export the Sanity dataset.
- If production issues occur, redeploy the last Shopify-backed storefront build or flip the temporary blog source switch, then fix Sanity content/migration offline.

</rollback_strategy>
