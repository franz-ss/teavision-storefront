# Phase 16 - UI Design Contract

> Visual and interaction contract for Phase 16 legal, consent, analytics, and SEO launch coverage. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Manual Teavision Tailwind 4 token system; shadcn is not initialized because `components.json` is absent and the repo already has approved tokens/primitives. |
| Preset | Not applicable. |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, checkbox/radio inline gaps, compact metadata spacing. |
| sm | 8px | Banner action gaps, stacked preference-row internals, compact status chips. |
| md | 16px | Default form-field and card content spacing. |
| lg | 24px | Policy-page content groups, modal body padding, banner text/action separation. |
| xl | 32px | Page section clusters, approval-matrix groups, cookie-preference panels. |
| 2xl | 48px | Major legal-page breaks and consent-page top/bottom rhythm. |
| 3xl | 64px | Page-level separation when a legal route has multiple major sections. |

Exceptions: Do not add new spacing tokens. Interactive touch-target sizing must come from existing `Button` and form primitives such as `min-h-11` or `min-h-12`. Existing responsive layout tokens `px-gutter`, `py-section`, `max-w-prose`, `max-w-base`, and `max-w-wide` remain approved because they are already defined in `src/app/globals.css`.

## Typography

Use exactly these four size roles and exactly these two declared weights on new Phase 16 UI surfaces:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.6 |
| Label | 14px | 500 | 1.4 |
| Heading | 24px | 500 | 1.2 |
| Display | 32px | 500 | 1.1 |

Implementation notes:

- Prefer existing utilities that map to the approved scale: `type-body`, `type-body-sm`, `type-heading-05`, and `type-heading-02`.
- Do not introduce additional local font-size utilities for consent, legal, analytics, or SEO evidence UI.
- Long legal copy uses `max-w-prose`, `type-body`, and `text-ink-soft`; section headings use display font with calm hierarchy rather than promotional hero sizing.
- Button labels use specific verb+noun copy; do not create giant or viewport-scaled CTA text.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-paper` / `--color-paper` | Page backgrounds and primary legal/cookie-preference surfaces. |
| Secondary (30%) | `bg-card`, `bg-paper-2`, `border-hairline` | Cards, legal approval matrix rows, modal panels, banner containers, footer/legal link grouping. |
| Accent (10%) | `text-brand`, `bg-brand`, `ring-ring` | Primary consent CTA, checked consent category controls, active preference state, focus ring, selected route-matrix state only. |
| Notice | `bg-gold-tint`, `text-gold-deep`, `border-gold-deep/40` | Pending owner/legal review banners and owner-gated verification callouts only. |
| Destructive | `text-danger`, `bg-danger-tint`, `border-danger/30` | Irreversible privacy/data-deletion request affordances and destructive confirmation messaging only. |

Accent reserved for: `Accept optional cookies`, `Save consent preferences`, checked `analytics` and `marketing` preference controls, active `Cookie preferences` link state, and focus rings. Accent must not be applied to every link or all interactive elements; ordinary policy links use `text-brand` only where the existing rich-text pattern already does so.

## Visual Hierarchy And Interaction Surfaces

| Surface | Primary Focal Point | Hierarchy Contract |
|---------|---------------------|--------------------|
| Static legal/policy pages | Policy title plus visible review banner when approval is pending. | The review banner appears above body copy on pending pages; it must be readable as a status notice, not a decorative badge. Body content stays prose-width. Related links and contact CTAs sit after the legal substance. |
| Cookie preferences page | Consent category controls and `Save consent preferences`. | Essential is visibly locked/on. Analytics and marketing are separate controls with short plain-language descriptions. Save action is the only brand-filled CTA. |
| First-visit consent banner | Three choices: accept optional, reject optional, manage preferences. | Banner is compact, bottom anchored or otherwise non-blocking on mobile, and does not hide cart/checkout primary actions. `Manage preferences` opens the modal/page flow. |
| Preference modal | Dialog title, category controls, then save/reject actions. | Use existing `Dialog`; trap focus, restore focus on close, support Escape close, and keep content within `max-h-[92vh]`. |
| Legal approval matrix/evidence | Approval state and proof link/status. | Status chips use semantic notice/destructive/brand states; do not rely on color alone. Matrix rows must remain scannable on mobile. |

Icon-only actions: allowed only for dialog close or compact controls using existing `IconButton`; every icon-only control needs an `aria-label` and visible focus ring.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Save consent preferences |
| Consent banner accept | Accept optional cookies |
| Consent banner reject | Reject optional cookies |
| Consent banner manage | Manage preferences |
| Legal review banner heading | Pending owner/legal review |
| Legal review banner body | This policy route is live for launch review, but final wording still needs owner/legal approval before it is treated as final. |
| Empty state heading | No optional tracking is active |
| Empty state body | Analytics and marketing stay off until you choose to allow them. You can update this anytime from Cookie Preferences. |
| Error state | We couldn't save your cookie preferences. Try again, or email info@teavision.com.au for help. |
| Destructive confirmation | Data deletion request: show "This request asks Teavision to permanently erase account and personal data where legally possible. This cannot be undone." before opening or submitting the request. Reject optional cookies: no destructive confirmation because the choice is reversible from Cookie Preferences. |

Additional copy rules:

- Do not use generic CTA labels such as "Submit", "OK", "Cancel", "Click Here", or "Save".
- Policy pages must not invent final legal promises. Pending pages show review status and route users to contact/support for clarifications.
- Analytics/SEO evidence surfaces use operational labels such as "Verify launch indexing", "Record destination check", and "Open route evidence" only when those actions exist.

## Component And Layout Contract

| Need | Approved Pattern |
|------|------------------|
| Page-level legal sections | `Section.Root` plus `Section.Container`; no raw page-level `section` elements outside the primitive. |
| Legal prose | Prose-width wrapper using existing rich-text/link/focus patterns from the current terms content. |
| Pending-review banner | `Card` or route-local notice component with `gold` semantic colors; visible before policy body. |
| Consent banner | Client leaf only; use `Button`, compact text, and explicit Accept/Reject/Manage actions. |
| Preference modal | Existing `Dialog`; category rows use `Checkbox` or `Radio` where semantics match; no parent wrapper `'use client'`. |
| Cookie preferences route | Server-rendered page shell with a client preference leaf; canonical URL `/pages/cookie-preferences`. |
| Footer legal links | Drive from central footer/link data or legal registry; labels must include Privacy Policy, Shipping Policy, Refund Policy, Terms of Service, and Cookie Preferences. |
| Evidence/status displays | Tables or definition-list style layouts with `Badge`/semantic text labels; do not make color the only state indicator. |

Implementation must keep Tailwind token classes, `cn()` class composition, one component per file once extracted, and Storybook stories for any reusable component placed in `src/components/`.
