In `callback/route.ts`, build every success and failure redirect from `getCustomerAccountRedirectOrigin()`, not from `request.url`. Keep callback validation fail-closed: bad/missing code, state, nonce, or token exchange clears pending auth, sets the safe flash message, and redirects to `reason=verification-failed`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> login start route | Untrusted returnTo and forwarded-origin headers enter OAuth start. |
| Shopify -> callback route | Untrusted callback params enter state/nonce/token validation. |
| account session -> Customer Account API | Customer access token crosses into Shopify GraphQL transport. |
| Customer Account API -> UI mappers | External customer/order/address data enters app-facing account types. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-06-01 | Spoofing | login/start route | mitigate | Canonicalize OAuth start to `SHOPIFY_CUSTOMER_ACCOUNT_REDIRECT_URI` origin before writing pending auth state. |
| T-14-06-02 | Tampering | callback route | mitigate | Validate code, state, and nonce before session creation; clear pending auth on failure. |
| T-14-06-03 | Information Disclosure | customer-account/client.ts | mitigate | Send raw token only in server-side Authorization header and preserve token redaction in errors. |
| T-14-06-04 | Denial of Service | fake Customer Account API | accept | Fake server rejects stale schema in tests only; live availability remains Shopify-owned. |
| T-14-06-05 | Elevation of Privilege | account operations | mitigate | Keep all reads/mutations session-scoped and do not accept client-provided customer IDs. |
</threat_model>
