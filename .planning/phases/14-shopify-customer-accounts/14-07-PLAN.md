<objective>
Close the profile editing UAT gap by making phone read-only instead of implying Shopify can persist it.

Purpose: Shopify Customer Account API `CustomerUpdateInput` currently supports only first and last name. The UI and Server Action must stop accepting phone as writable while still showing the account-managed phone value when available.

Output: Profile input types/actions that ignore phone writes, profile UI that renders phone read-only like email, and tests/stories proving no editable phone control exists.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form -> Server Action | Untrusted profile form data crosses into authenticated mutation boundary. |
| Server Action -> Customer Account API | Writable fields are sent to Shopify's customerUpdate mutation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-07-01 | Tampering | updateProfileAction | mitigate | Ignore stale `phone` form data and construct mutation input from allowlisted writable fields only. |
| T-14-07-02 | Information Disclosure | ProfileForm | accept | Phone is already returned by authenticated Customer Account profile; read-only display is scoped to signed-in account pages. |
| T-14-07-03 | Repudiation | ProfileForm | mitigate | UI copy makes phone account-managed, avoiding false success states for unsupported writes. |
| T-14-07-04 | Elevation of Privilege | updateProfileAction | mitigate | Preserve `requireCustomerAccountSession('/account/profile')` before any mutation. |
</threat_model>
