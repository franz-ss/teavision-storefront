<threat_model>

## Trust Boundaries

| Boundary | Description |
| --- | --- |
| Local lab metrics to launch decision | Local Lighthouse failures can only become non-blocking through strict pass evidence or dated acceptance |
| Human acceptance to automated score | Acceptance must be explicit, dated, and validated before it changes the final audit status |
| Owner-gated proof to code readiness | Shopify/admin/Search Console gates stay separate from automated code readiness |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
| --- | --- | --- | --- | --- |
| T-17-29 | Repudiation | performance acceptance | mitigate | Require Date, Approver, Evidence source, and explicit route decisions |
| T-17-30 | Tampering | final readiness score | mitigate | Keep performance FAIL blocking unless strict pass or valid acceptance exists |
| T-17-31 | Spoofing | owner/staging approval | mitigate | Do not invent approver names, evidence links, or acceptance dates |
| T-17-32 | Information Disclosure | launch reports | mitigate | Preserve existing redaction and do not include secrets, tokens, PII, checkout URLs, cart IDs, or raw provider payloads |

</threat_model>
