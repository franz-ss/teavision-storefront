<locked_decisions>
- D-01 through D-07: public `?page=N` URLs and production canonical/crawler behavior are launch requirements. D-05 (refined): prev/next via hoisted `<link>` tags — the Next 16 Metadata API cannot express them.
- D-08 through D-13: Shopify Storefront GraphQL remains source of truth; keep **product payloads** bounded and do not widen listing payloads. D-10 (refined): page→cursor resolution uses an id-only cursor index query (NOT sequential cursor-walking); id/cursor-only requests up to `first: 250` are explicitly allowed.
- D-14 through D-16, D-23 through D-26: render the full classic pager with true last page (totals come from the cursor index), via a shared `src/components/ui/pagination/` primitive; sort/filter changes reset to page 1; pager clicks scroll to grid top; match Phase 11 PLP/search styling.
- D-17 through D-20: tests and route verification are mandatory.
- D-21: page size stays 24 (production's 48/page surfaced and explicitly declined).
- D-22, D-24: out-of-range AND stale-cursor pages redirect to the last valid page — never render an empty 200 listing for a bad page param.
- D-27: category pages canonicalize to the parent collection URL (fixes the current missing-canonical gap with production parity).
</locked_decisions>

## Task 1: Replace public cursor state with page-number state

type: implementation

files:

- `src/app/(storefront)/collections/[handle]/_lib/page-types.ts`
- `src/app/(storefront)/collections/[handle]/_lib/page-helpers.ts`
- `src/app/(storefront)/collections/[handle]/_lib/page-helpers.test.ts`
- `src/app/(storefront)/collections/[handle]/_components/page-content.tsx`

action:

- Replace route-level public `cursor` handling with `page`.
- Add a helper such as `parsePageParam(value): number` that returns `1` for missing/invalid values and never returns less than `1`.
- Update href generation helpers so page 1 omits `?page=1` and page 2+ uses `page=N`.
- Pager hrefs preserve current `sort` and allowed `filter` query params exactly as current helpers do.
- Sort and filter hrefs DROP the `page` param entirely (changing sort/filter always returns to the clean page-1 URL) per D-25.
- Do not expose internal cursor values in generated links.

verify:

- Unit tests cover missing, invalid, zero, negative, decimal, and large `page` params.
- Unit tests cover href generation for base collections, category collections, sorted collections, and filtered collections.
- Unit tests cover the page-reset rule: a sort/filter href generated while on `?page=5` contains no `page` param.

acceptance_criteria:

- Product list pagination links use `?page=N`.
- No normal PLP UI path emits `?cursor=`.

## Task 2: Add cursor-index page resolution with true totals

type: implementation

files:

- `src/lib/shopify/operations/collection.ts`
- `src/lib/shopify/queries/collection.graphql`
- `src/lib/shopify/types/index.ts`

action:

- Add an **id-only cursor index query** to `collection.graphql`: fetch only `edges { cursor }` (plus `pageInfo { hasNextPage endCursor }`) with `first: 250` — NO product fields. If `hasNextPage` is true after 250, continue in 250-cursor chunks (1 request per 250 products; `/collections/all` has ~230 today).
- Add a helper such as `getCollectionPageIndex()` that returns total product count, true total pages for a given page size, and the `after` cursor for any page N (the cursor at position `(N-1) * pageSize - 1`).
- Cache the index helper with `use cache`, `cacheTag('collection', \`collection-${handle}\`)`, and `cacheLife('hours')` — identical policy to the product fetch so index and page data cannot diverge by more than one cache window (D-11).
- Add a helper such as `getCollectionProductsPage()` that accepts `handle`, `page`, `first`, `sortKey`, `reverse`, and `filters`; it resolves the page cursor via the index, then runs the existing bounded product query for just that page.
- Extend collection product pagination data if needed with `startCursor`/`hasPreviousPage`, but do not add unrelated product fields.
- Return the requested page products, pagination state, and true `totalPages`/`totalCount` for the UI.
- Keep `COLLECTION_PRODUCT_PAGE_SIZE = 24` (D-21 — production's 48/page was surfaced and explicitly declined).

verify:

- Tests or focused mocks prove page 1 renders with one index request plus one bounded product query (both cached).
- Tests or focused mocks prove page N never sequentially walks product-payload requests and never requests product fields beyond the single 24-item page.
- Tests prove the index helper computes correct totals and per-page cursors from mocked cursor lists, including the >250 chunking path.
- If GraphQL changes, run `pnpm codegen`.

acceptance_criteria:

- Requesting page 2 renders the second product page using Shopify cursor data internally.
- The UI receives a true total page count derived from the index.
- Product card payload remains minimal and compatible with quick-add behavior.

## Task 5: Handle invalid and out-of-range pages safely

type: implementation

files:

- `src/app/(storefront)/collections/[handle]/_components/page-content.tsx`
- `src/app/(storefront)/collections/[handle]/_lib/page-helpers.ts`
- `src/lib/shopify/operations/collection.ts`

action:

- Treat invalid page values as page 1 and generate clean page-one links.
- Out-of-range pages (`page > totalPages` from Task 2's index) **redirect to the last valid page** (D-24) — totals are reliable now, so no `notFound()` and never an empty indexable 200 listing. (Production renders 200-empty here; intentionally not copied.)
- Stale cached cursors (page fetch returns empty/short despite `page <= totalPages`) reuse the same redirect path (D-22) — no force-refresh or retry machinery.
- Do not break genuine empty filtered results; keep the existing empty-state behavior for valid filter combinations with no matches.

verify:

- Tests cover invalid page values, an out-of-range redirect, and the stale-cursor fallback path.
- Existing empty-state tests or fixtures still pass.

acceptance_criteria:

- Bad pagination parameters do not create soft-404 indexable pages.
- Valid no-result filters still render the intended no-match UI.
