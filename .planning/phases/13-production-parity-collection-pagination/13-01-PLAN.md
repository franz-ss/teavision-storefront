## Task 1: Replace public cursor state with page-number state

type: implementation

files:

- `src/app/(storefront)/collections/[handle]/_lib/page-types.ts`
- `src/app/(storefront)/collections/[handle]/_lib/page-helpers.ts`
- `src/app/(storefront)/collections/[handle]/_components/page-content.tsx`

action:

- Replace route-level public `cursor` handling with `page`.
- Add a helper such as `parsePageParam(value): number` that returns `1` for missing/invalid values and never returns less than `1`.
- Update href generation helpers so page 1 omits `?page=1` and page 2+ uses `page=N`.
- Preserve `sort` and allowed `filter` query params exactly as current helpers do.
- Do not expose internal cursor values in generated links.

verify:

- Unit tests cover missing, invalid, zero, negative, decimal, and large `page` params.
- Unit tests cover href generation for base collections, category collections, sorted collections, and filtered collections.

acceptance_criteria:

- Product list pagination links use `?page=N`.
- No normal PLP UI path emits `?cursor=`.

## Task 2: Add bounded page-to-cursor resolution

type: implementation

files:

- `src/lib/shopify/operations/collection.ts`
- `src/lib/shopify/queries/collection.graphql`
- `src/lib/shopify/types/index.ts`

action:

- Extend collection product pagination data if needed with `startCursor`/`hasPreviousPage`, but do not add unrelated product fields.
- Add a helper such as `getCollectionProductsPage()` that accepts `handle`, `page`, `first`, `sortKey`, `reverse`, and `filters`.
- Resolve page N by walking Storefront `pageInfo.endCursor` in bounded requests until the requested page cursor is known.
- Cache the helper with `use cache`, `cacheTag('collection', \`collection-${handle}\`)`, and the current collection cache lifetime.
- Return only the requested page products and pagination state needed by the UI.
- Keep `COLLECTION_PRODUCT_PAGE_SIZE = 24` unless implementation evidence shows production uses a different page size that the owner explicitly wants.

verify:

- Tests or focused mocks prove page 1 uses one product query.
- Tests or focused mocks prove page N uses bounded page-size requests and does not request 250 products or all products.
- If GraphQL changes, run `pnpm codegen`.

acceptance_criteria:

- Requesting page 2 renders the second product page using Shopify cursor data internally.
- Product card payload remains minimal and compatible with quick-add behavior.

type: implementation

verify:

- Tests assert current page state, previous/next links, and page links.
- Visual structure uses warm/botanical token classes only.

acceptance_criteria:

- PLPs present classic page-number navigation.
- Pagination works without layout shifts or text overflow at mobile and desktop widths.

## Task 4: Preserve production metadata and crawler behavior

type: implementation

files:

- `src/app/(storefront)/collections/[handle]/page.tsx`
- `src/app/(storefront)/collections/[handle]/[category]/page.tsx`
- `src/app/(storefront)/collections/[handle]/_lib/page-helpers.ts`

action:

- Add `alternates` or metadata fields needed for canonical output.
- Add adjacent prev/next link metadata where Next supports it cleanly; if metadata API cannot express it, document the limitation and use route-rendered head/link strategy only if compatible with Next 16 App Router.
- Normalize `?page=1` to the clean URL in generated links. Add redirects only if safe and easy in the existing route structure.

verify:

- Tests or route HTML checks confirm canonical for `/collections/all?page=2` points at `/collections/all`.
- Route checks confirm prev/next for page 1 and page 2 where emitted.
- Route checks confirm sort/noisy filter URLs are not made more indexable than production.

acceptance_criteria:

- SEO behavior matches production for launch.
- No new indexable cursor URLs are emitted.

## Task 5: Handle invalid and out-of-range pages safely

type: implementation

files:

- `src/app/(storefront)/collections/[handle]/_components/page-content.tsx`
- `src/app/(storefront)/collections/[handle]/_lib/page-helpers.ts`
- `src/lib/shopify/operations/collection.ts`

action:

- Treat invalid page values as page 1 and generate clean page-one links.
- For out-of-range pages that can be detected, prefer `notFound()` or a redirect to the last known valid page over rendering an empty indexable listing.
- Do not break genuine empty filtered results; keep the existing empty-state behavior for valid filter combinations with no matches.
- Decide out-of-range behavior based on available pageInfo evidence and test it.

verify:

- Tests cover invalid page values and an out-of-range scenario.
- Existing empty-state tests or fixtures still pass.

acceptance_criteria:

- Bad pagination parameters do not create soft-404 indexable pages.
- Valid no-result filters still render the intended no-match UI.
