## Why gaps_found Despite a Passed Phase Verification

Phase verification confirmed the **server-rendered** href generation honors D-25 (sort/filter links drop `page`). The integration check examined the **client components** that actually fire on user interaction and found both bypass that logic:

1. **`SortSelect.createSortUrl`** ([sort-select.tsx:26-33](../src/components/collection/sort-select/sort-select.tsx)) — clones `useSearchParams()`, sets/deletes `sort`, never deletes `page`. Changing sort on `/collections/all?page=3` yields `/collections/all?sort=title-asc&page=3`.
2. **`FilterPanel.replaceFilters`** ([filter-panel.tsx:39-47](../src/components/collection/filter-panel/filter-panel.tsx)) — clones `useSearchParams()`, replaces `filter` entries, never deletes `page`. Toggling a checkbox filter on page > 1 keeps the stale page.

Consequence: the user lands on page N of the re-sorted/re-filtered result set (or hits the out-of-range redirect if the new set is shorter) instead of resetting to page 1 as D-25 specifies and production behaves. Safety rails (out-of-range redirect, canonical-to-base) prevent SEO junk, so this is a **behavioral parity gap, not an SEO hazard** — but it is a direct violation of locked decision D-25 and of PLP-PAGE-05's "sort/filter semantics are preserved."

**Fix size:** one `params.delete('page')` call in each component's URL-building function, plus test coverage. Small enough for a gap plan rather than a full new phase.
