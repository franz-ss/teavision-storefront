# Crawlability: Server-Rendered HTML for Collection/Product Pages

**Domain:** Next.js 16 fork, Cache Components (`cacheComponents: true`), App Router, Shopify headless storefront
**Researched:** 2026-07-10
**Overall confidence:** HIGH (fork docs + code inspection; one MEDIUM item on real-world bot behavior)

---

## Streaming & Crawlers (fork-verified)

**Verified against:** `node_modules/next/dist/docs/01-app/02-guides/streaming.md`, `.../01-app/01-getting-started/08-caching.md`, `.../01-app/02-guides/rendering-philosophy.md`.

1. **Streaming is real chunked HTTP, one response.** React sends a "static shell" (everything above/outside Suspense boundaries, including fallbacks) in the first chunk, then streams resolved boundary content as additional chunks in the *same* HTTP response (`streaming.md` "The HTML stream" / "The static shell"). Technically, a client that downloads and waits for the full response body eventually receives full markup either way.

2. **`fallback={null}` is a genuine crawlability defect in this codebase, not a false alarm** — for two independently fork-documented reasons:
   - **The fork itself special-cases non-JS bots.** `streaming.md` § "Metadata and bots": *"`generateMetadata` resolves before streaming begins for bots that only scrape static HTML (such as Twitterbot or Slackbot)... Next.js automatically detects user agents to choose the right behavior."* This is explicit confirmation that the framework's authors know some crawlers only read the static shell and never process the streamed chunks/swap scripts (`htmlLimitedBots` config exists specifically for this). The same asymmetry applies to page **content**, not just metadata: a non-JS-executing fetch of the raw HTML (curl-based SEO auditing tools, many non-Google crawlers, link-preview bots, and — per the milestone's own consultant finding — whatever crawlability check flagged this) sees only the shell. With `fallback={null}` wrapping the entire page, that shell is empty.
   - **`fallback={null}` here isn't a stylistic choice — it's forced by the current architecture.** Both `collections/[handle]/page.tsx` and `products/[handle]/page.tsx` `await searchParams` (sort/page/filter, and `variant`, respectively) unconditionally at the top of the async component that renders everything. Under `cacheComponents`, `searchParams` is a runtime-only API — per `08-caching.md` § "Working with runtime APIs" and the project's own recorded finding (`project_cachecomponents_params_suspense` memory), any component that awaits `searchParams` **must** be wrapped in `<Suspense>` or the build fails with the "blocking route" error. Route `params` do **not** carry this requirement (confirmed both in the fork's caching guide and in this project's own memory: params can be read outside Suspense, even for values `generateStaticParams` didn't pre-list). So today's `<Suspense fallback={null}><PageContent .../></Suspense>` exists solely to satisfy the `searchParams` constraint — and because the `searchParams` await sits at the very top of the tree, *everything* downstream (H1, hero, description, first product-grid rows) got swept into that one boundary and its empty fallback.

3. **This is exactly the class of defect the blog fix already eliminated** (see "Reference: Blog Fix Generalized" below) by never awaiting `searchParams` in the indexable listing at all.

**Verdict:** treat this as a real, fork-confirmed defect. Fix = stop letting a `searchParams` read at the top of the tree force the whole page into one empty-fallback Suspense boundary.

## Target Render Pattern

Confirmed pattern from `streaming.md` § "Push dynamic access down" (this is the fork's own prescribed fix, verbatim): *"The key to maximizing what streams instantly is to defer dynamic data access to the component that actually needs it... If you `await` any of these [params, searchParams, cookies, headers, data fetches] at the top of a layout or page, everything below that point becomes dynamic."* The doc's own worked example (`app/shop/[category]/page.tsx`) is structurally identical to this project's collection page: `<Hero />` renders outside Suspense; `<ProductGrid paramsPromise={params}>` is the only thing inside Suspense.

Generalized pattern for both routes:

1. **Do not await `searchParams` (or `variant`/`sort`/`page`/`filter`) at the top of the page/content component.** Keep the outer async function driven only by route `params` (which — with full `generateStaticParams` coverage on both routes — resolves at build time and needs no Suspense at all).
2. **Hoist the default-view data fetch above Suspense.** Both routes already have a fully cache-key-deterministic "canonical" fetch that depends only on `handle` (+ `category` param for collections): `getProduct(handle)` / `getCollection(handle)` + `getCollectionProductsPage(handle, 1, PAGE_SIZE, defaultSort, false, [])`. These are `'use cache'`-backed (verified in `src/lib/shopify/operations/product.ts` and `collection.ts`) and produce identical output whenever the request has no non-default `searchParams` — which is the case for every canonical URL a crawler follows (internal links, sitemap, and the page's own `<link rel="canonical">`, which already always points at the base collection/product URL per `getPath()`/`generateMetadata`). Render H1, breadcrumbs, hero, JSON-LD, description, and the first product-grid rows straight from this fetch, with **no Suspense wrapper** — it becomes part of the prerendered static shell.
3. **Narrow the Suspense boundary to just the searchParams-dependent slice**, and — this is the non-obvious but doc-supported part — **use the same default-view render as the Suspense `fallback`, not a skeleton.** Suspense fallbacks are arbitrary JSX (`streaming.md`: "The fallback... is included in the static shell"); nothing requires them to be loading spinners. Pass the already-rendered default `ProductList`/`Toolbar` (computed in step 2) as `fallback` to a `<Suspense>` wrapping a small `<FilteredResults searchParams={searchParams} .../>` component that awaits `searchParams`, compares against the defaults, and only re-fetches/re-renders when the user actually requested a non-default sort/filter/page. Net effect:
   - Canonical URL (no query string) → static shell already shows the *correct* content; the streamed replacement (once it resolves) renders identically, so there's no visible swap and no wasted work for the common case.
   - `?sort=price-asc` etc. → shell shows the default view for an instant, then streams in the corrected view. This only affects the minority of requests that pass non-default query params — URLs that are non-canonical anyway per the existing `<link rel="canonical">`.
4. **Product page: move the `variant` query param off the server render path entirely.** `initialVariantId` only affects which variant `ProductForm` (already `'use client'`) preselects — it is not SEO text content. Per `use-search-params.md` § "Server Components", read `variant` via the Client Component's own `useSearchParams()` instead of via the server `searchParams` prop, wrapped in a **local** `<Suspense>` scoped to just `<ProductForm>` (required by the fork: *"a static page that calls `useSearchParams` from a Client Component must be wrapped in a Suspense boundary, otherwise the build fails"*), with the fallback being `<ProductForm>` rendered with its ordinary default-variant selection (no `initialVariantId`). This removes the *only* reason `ProductContent` currently awaits `searchParams` at all, letting the entire rest of the PDP (gallery, title, description, disclosures, reviews section, tags) become static-shell content driven purely by `params.handle`.
5. **Keep `RelatedProducts` / `CustomersAlsoBought` in their existing nested `<Suspense fallback={null}>`.** They take an already-resolved `product` object as a prop and touch neither `params` nor `searchParams` (verified: no references in either file) — this is legitimate below-the-fold streaming of slow/uncached work, exactly the "Streaming uncached data" pattern from `08-caching.md`, and is unaffected by (in fact benefits from) removing the outer blocking boundary.

## Cache Components Interaction

- **`'use cache'` functions are what make step 2 possible.** `getProduct`, `getCollection`, `getCollectionProductsWithFilters` (and the cursor-index helpers `getCollectionProductsPage` sits on top of) are already `'use cache'`. A cached, deterministic, params-only-keyed call is eligible to run *during prerendering* and be baked into the static shell (`08-caching.md` § "How rendering works": *"`use cache`: the result is cached and included in the static shell"*). No caching changes are required — the defect is purely about **where in the component tree** these already-cached calls are awaited, not whether they're cached.
- **`generateStaticParams` coverage is already sufficient for a real static shell.** `products/[handle]/page.tsx` calls `getAllProducts()` and `collections/[handle]/page.tsx` calls `getCollections()` — both routes are fully enumerated ahead of time, so `params` never needs a Suspense boundary and every canonical product/collection URL is eligible for build-time PPR. `category` is a route param too (not covered by `generateStaticParams`, since it only returns `{ handle }`), but per the fork's docs and this project's own verified finding, un-listed param combinations still render synchronously outside Suspense (served on-demand / ISR-style) — they are not forced dynamic the way `searchParams` is.
- **`searchParams` has no `generateStaticParams` equivalent and never resolves at build time** — any component awaiting it is unavoidably dynamic and must live inside `<Suspense>`. This is a hard framework constraint, not a workaround; the fix is scoping *which* component does that await, not eliminating the await.
- **Cache tags are untouched by this refactor.** `cacheTag`-bearing operations keep their existing tags/`cacheLife` profiles; this is a component-tree/Suspense-placement change, not a data-layer or revalidation change. No `updateTag`/`revalidateTag` call sites need to change.
- **Root layout constraint (relevant sanity check, not a needed change):** `08-caching.md` § "Opting out of the static shell" warns that a `<Suspense fallback={null}>` directly around `<body>` in the **root layout** kills the static shell for the *entire app*. That is not what's happening here — the empty-fallback Suspense boundaries in question are inside individual `page.tsx` files, not the root layout — but it's worth confirming `src/app/(storefront)/layout.tsx` (and the root `src/app/layout.tsx`) don't have a similar body-level Suspense wrapper, since that would silently defeat any per-page fix made here.

## Reference: Blog Fix Generalized

`src/app/(storefront)/blogs/[blog]/page.tsx` (commit `28b9f886`) is the working template, and the mechanism generalizes directly:

| | Blog listing (shipped) | Collection/product (target) |
|---|---|---|
| Route params | `blog` (fully covered by `generateStaticParams`) | `handle` (+ `category` for collections) — fully/partially covered |
| Query-string state | **None** — pagination moved to path segments (`/blogs/[blog]/page/N`), filtering/search moved to a **separate route** (`/blogs/[blog]/search`) | Sort/filter/page are still `?searchParams` on the same route |
| Suspense usage | **Zero** — `BlogPage` is a plain `async function`, no `<Suspense>` anywhere | Currently one `<Suspense fallback={null}>` wrapping everything |
| Data fetch | `getDefaultBlogListing(blog, 1)` — cached, params-only | `getCollection`/`getProduct` + page-1 default product fetch — already cached, params-only, just not hoisted |
| Result | Entire listing (including "Read more" disclosures) is static-shell HTML | Not yet — blocked by the `searchParams` await sitting above everything |

The blog fix took the more aggressive route (searchParams-driven behavior banished to a dedicated URL entirely), which is why it needed *no* Suspense at all. For collections/products, moving sort/filter/pagination to dedicated routes/paths is a larger, likely out-of-scope refactor for Phase 27 (it changes public URLs and pagination UX). The **narrower, in-scope equivalent** is the "hoist the default fetch, shrink the Suspense boundary, use the default render as the fallback" pattern in "Target Render Pattern" above — it achieves the same crawlability outcome (real content in the initial shell for every canonical URL) without touching the URL scheme. If a future milestone wants full parity with the blog approach, moving collection pagination from `?page=N` to `/collections/[handle]/page/[page]` (mirroring the blog's `/page/N` migration) and moving sort/filter to a client-side-only refinement UI would let the collection page drop Suspense entirely too — flag this as a possible Phase 27+1 follow-up, not required now.

## Pitfalls & Prevention

- **Prior known bug — multiple/changing H1s from `cacheComponents` Activity DOM retention** (project memory `project_seo_h1_cachecomponents`): `cacheComponents: true` keeps up to 3 previously-visited routes mounted in hidden `<Activity mode="hidden">` containers after **client-side soft navigation**, so DOM-reading crawlers/extensions can see leaked H1s from prior routes in the live browser DOM. This is a *separate* defect from the one being fixed here (this phase is about the raw server HTML of a single fresh request/URL, not multi-route client DOM accumulation) but the fix in this phase must not accidentally make it worse: don't introduce new client-side route transitions or additional `<h1>`-bearing components that get soft-navigated between. The known mitigation (hard navigation via plain `<a href>` on SEO-critical links) is unrelated to this phase's scope and should not be conflated with it. Regression-test guidance from that memory still applies if any new navigation surfaces are touched: check raw `page.locator('h1').count()` after soft nav, not `getByRole('heading')`.
- **LCP regression risk.** Per `streaming.md` § "LCP": *"If your LCP element... is inside a Suspense boundary, it can't paint until that boundary resolves. Keep LCP elements outside or above Suspense boundaries."* The product page's LCP element is almost certainly the gallery hero image or the H1; both must end up in the un-Suspended, params-only portion of the tree (step 2 above) — do not leave the gallery/title inside the narrowed Suspense boundary meant only for `ProductForm`'s variant selection. Same for the collection page's hero image/H1 relative to the narrowed product-grid Suspense.
- **CLS risk from swapping the "default view" fallback → "actual view" content.** Because the recommended pattern reuses the default render as the Suspense fallback (not a skeleton), sizes should already match closely for the common (canonical) case — but for non-default `searchParams` requests (different sort/filter/page), the swap-in content can have a different item count/height than the fallback. Per `streaming.md` § "CLS": size the grid container with a `min-height` or reserve grid rows so the swap doesn't visibly reflow.
- **Over-fetching from hoisting awaits.** Hoisting the "default view" fetch above Suspense means it now runs unconditionally on every render pass of that route (as part of the static shell / prerender), even for requests that immediately override it with non-default `searchParams`. This is intentional and cheap here because the calls are `'use cache'`-backed (Shopify round-trip only happens on cache miss/revalidation, not per-request) — confirm this stays true and don't accidentally strip `'use cache'` from `getCollectionProductsPage`'s underlying calls while refactoring where they're invoked from.
- **Don't reintroduce the `searchParams`-forces-Suspense problem one level down.** When narrowing the boundary, make sure the *only* component awaiting `searchParams` is the small leaf inside the new inner `<Suspense>` — if any shared helper (e.g. `parseSelectedFilterParams`, `SORT_MAP` lookups) gets called from the outer/static portion using unresolved `searchParams`, the whole tree collapses back into one dynamic boundary.
- **`notFound()` / redirect timing.** `PageContent` currently calls `notFound()` and `redirect()` (out-of-range page, stale-cursor fallback) *after* fetching collection data that also depends on the (now-relocated) `searchParams`-derived filters. Per `streaming.md` § "When does streaming start?": `notFound()`/`redirect()` only produce a real HTTP status if they fire **before** any `await`/Suspense boundary starts streaming. Keep the `notFound()` guard for "collection handle doesn't exist" in the params-only, pre-Suspense portion (it already is, since `getCollection(handle)` doesn't need searchParams) so unknown collection handles still get a genuine 404 rather than a soft/client redirect. Page-out-of-range and stale-cursor redirects are inherently `searchParams`-dependent (they depend on `page`) and are correctly a client-side redirect once inside the inner Suspense — no regression there, just don't move the handle-existence check into the dynamic slice.
- **Verify no reverse-proxy/CDN buffering hides the improvement.** Per `streaming.md` § "What can affect streaming", the benefit of a smaller/faster static shell also requires the deployment platform to not buffer the whole response before forwarding. Not a code change, but worth a build-output / production check (`next build`'s static/PPR route table, and viewing page source directly) to confirm the shell actually contains the expected H1/content post-fix, per the doc's own verification tip: *"see what content was added to the static shell of any page by viewing the page source in your browser."*
